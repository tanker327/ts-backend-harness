# ADR-032: CORS Policy Deferred Until a Browser Client Exists

- **Status**: Accepted
- **Date**: 2026-04-17
- **Enforced by**: Absence of CORS middleware in `src/index.ts`; this ADR

## Context

The app is a Hono-based JSON API. In its current form it serves:

1. The OpenAPI spec at `/openapi.json` and Swagger UI at `/docs` (developer tooling on the same origin).
2. Better Auth at `/api/auth/**` (ADR-012) — which has its own origin configuration.
3. Business routes consumed by server-to-server callers and tests via `app.request()` (ADR-017).

There is no browser SPA coupled to this API today. The OpenAPI client is generated by hey-api (ADR-007) and consumed from frontends that are not yet in this repo. Adding a global CORS middleware now would force a guess at the allowed origin list before we know who the real caller is.

## Decision

Do **not** register global CORS middleware in `src/index.ts` at this time.

When the first browser-based client is introduced:

1. Add `hono/cors` middleware in `src/index.ts` **before** the auth handler and route registrations.
2. Pull the allowed-origin allowlist from env (validated in `src/config/env.ts` per ADR-008) — never hard-code.
3. Configure credentials-allowed only for origins that need it; keep the default `allowOrigin` list closed, not `*`.
4. Better Auth's `trustedOrigins` list is the source of truth for auth-endpoint origins — keep the CORS allowlist in sync.

This ADR exists so the gap is documented. An agent or human adding CORS must consult this ADR and supersede it rather than ad-hoc configuring the middleware.

## Consequences

### Positive
- No premature allowlist that rots or accidentally opens the API.
- Current server-to-server and test traffic is unaffected (no preflights, no header overhead).
- Decision is visible: future contributors see why CORS is absent and what to do when it's needed.

### Negative
- A browser client added without reading this ADR will hit silent CORS failures in the browser and may debug the wrong layer.
- When CORS is added, it must be coordinated with Better Auth's `trustedOrigins` — two lists to keep aligned.

### Enforcement
- Adding CORS middleware is an ADR trigger (ADR-002 — new cross-cutting pattern) and supersedes this ADR with a follow-up ADR describing the allowlist source and the environment-variable contract.
- Code review: a PR that introduces `hono/cors` must either supersede this ADR or be rejected.
