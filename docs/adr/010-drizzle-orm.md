# ADR-010: Drizzle ORM

- **Status**: Accepted (backfilled 2026-04-17; original adoption at project start, see ADR-001)
- **Date**: 2026-04-17
- **Enforced by**: `package.json`, `drizzle.config.ts`, `.claude/rules/database.md`, convention in `src/repos/`

## Context

The project needs a database access layer that:

- Produces type-safe queries without hand-writing SQL strings
- Plays well with TypeScript strict mode (ADR-004)
- Supports schema-as-code with a real migration story
- Stays close to SQL semantics (no leaky abstraction that breaks on non-trivial queries)

Candidates:

- **Prisma** — schema-as-DSL (`.prisma` file), generates a typed client. Excellent DX, but the schema lives outside TypeScript, requires a codegen step, and the query client has historically been heavy and less flexible for advanced SQL (raw JSON, window functions, CTEs).
- **Kysely** — pure TypeScript query builder. Excellent types, but no schema-as-code; you still write migrations by hand.
- **Drizzle** — schema defined in TypeScript (`pgTable`), types inferred directly, queries written in a SQL-shaped DSL, migrations generated by `drizzle-kit`. Lightweight and closer to SQL than Prisma.

For a template project that demonstrates best practices, Drizzle's "schema is just TypeScript" model is pedagogically cleaner — no separate schema language to learn, types flow naturally.

## Decision

Use **Drizzle ORM** for all database access. Its `drizzle-kit` tool handles migration generation and application.

- Schema files live in `src/types/` (or a dedicated schema directory colocated there)
- Queries live exclusively in `src/repos/` (see ADR-011)
- Migrations via `bunx drizzle-kit generate` + `bunx drizzle-kit migrate`
- Raw SQL is allowed only inside `src/repos/` functions and only when the query builder cannot express the query

## Consequences

### Positive
- Schema and types share one definition — no drift between DB and code
- Close to SQL — developers who know Postgres can read Drizzle queries at a glance
- Type inference catches most query mistakes at compile time
- Light runtime overhead

### Negative
- Younger ecosystem than Prisma — fewer blog posts and Stack Overflow answers
- Migration workflow (generate + migrate) has sharper edges than Prisma's push model
- Some advanced relations still require explicit type hints

### Enforcement
- `package.json` includes `drizzle-orm` and `drizzle-kit`
- `.claude/rules/database.md` states "Drizzle ORM only. No raw SQL unless wrapped in a repo function."
- Adding a second ORM or migration tool is an ADR trigger (ADR-002)
